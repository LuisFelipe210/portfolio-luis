DOCUMENTAÇÃO COMPLETA

Plataforma full-stack para venda de produtos físicos e agendamento de serviços técnicos em um fluxo único. Este documento cobre arquitetura, regras de negócio, segurança, APIs, banco, jobs, execução, testes e operação.

React + Vite Node + Express + Prisma MySQL 8 JWT + Refresh Rotativo + CSRF AES-256-GCM

1. Visão Geral

O sistema resolve um cenário híbrido:

  • Produto físico: catálogo, reserva de estoque, checkout, pedido.
  • Serviço: catálogo, seleção de slot, coleta de informações técnicas, checkout, pedido.

Perfis de acesso:

  • CUSTOMER: compra/agendamento e acompanhamento de pedidos.
  • VENDOR: operação do tenant (catálogo, inventário, status de pedidos, KPIs do próprio tenant).
  • ADMIN: visão global, gestão de tenants, curadoria da Home, KPIs globais.

2. Arquitetura Técnica

Frontend

React 19 + Vite + TypeScript + TanStack Query + React Router.

Backend

Node.js + Express + TypeScript + Prisma.

Banco

MySQL 8.

Infra local

Docker Compose (dev/prod).

Segurança e runtime:

  • Helmet, CORS estrito, rate limits, JWT + refresh token rotativo, CSRF token.
  • Criptografia AES-256-GCM para dados sensíveis de pedido.
  • node-cron para limpeza de reservas e reposição de slots.
U[Usuário] -> F[Frontend React] -> B[Backend Express] -> P[Prisma] -> D[(MySQL)]
B -> J1[cleanupReservations]
B -> J2[serviceSlotTopup]

4. Modelo de Dados (Prisma)

Entidades principais:

  • User: usuário com role (ADMIN|VENDOR|CUSTOMER), vínculo opcional com tenant, flag mustChangePassword.
  • Tenant: parceiro/empresa (loja/assistência).
  • Location: pontos físicos do tenant.
  • Category: categorias de catálogo com flags de destaque home.
  • Item: produto ou serviço com preço, categoria, tenant, custom fields e flags de destaque.
  • Inventory: estoque por par (itemId, locationId).
  • ServiceSlot: agenda de horários para itens do tipo SERVICE.
  • CartReservation: reserva temporária (soft reservation).
  • Order e OrderItem: pedido consolidado por tenant e suas linhas.
  • MaintenanceIntake: dados técnicos coletados para linhas de serviço.
  • RefreshToken: refresh token hasheado por usuário.

Regras estruturais:

  • Inventory possui @@unique([itemId, locationId]).
  • Order indexado por tenantId, status, createdAt.
  • RefreshToken.token é único e armazenado como hash SHA-256.

5. Segurança

5.1 Backend

  • Validação obrigatória de env críticos no startup.
  • Bloqueio de segredo fraco/chave inválida em produção.
  • helmet com frameguard deny, HSTS em produção, referrerPolicy no-referrer e CSP.
  • CORS allowlist por CORS_ORIGINS/FRONTEND_URL.
  • Body size limitado: 500kb.
  • Rate limit em camadas: global, auth, refresh, carrinho/reserva e checkout.
  • JWT HS256 curta duração + refresh rotativo.
  • CSRF em cookie + header para operações sensíveis (produção).
  • Origem confiável (Origin/Referer) em refresh/logout/change-password.
  • Revogação de sessões na troca de senha.
  • Criptografia AES-256-GCM para fulfillmentData.
  • Sanitização de dados em respostas de pedido por papel.

5.2 Frontend

  • Access token em memória (não persiste em localStorage).
  • Refresh mutex para evitar corrida de múltiplos 401 simultâneos.
  • CSRF token lido de cookie e enviado em mutações.
  • Sanitização de inputs com DOMPurify em formulários críticos.
  • CSP em frontend/index.html.

5.3 Pipeline de segurança

  • Semgrep (OWASP Top Ten + Secrets).
  • Gitleaks.
  • npm audit --omit=dev --audit-level=high backend/frontend.
  • Testes + build backend/frontend.

6. Autenticação e Sessão

6.1 Login/Register

  1. Frontend chama /api/auth/login ou /api/auth/register.
  2. Backend valida payload (Zod + política de senha).
  3. Gera token JWT (access) e refresh assinado salvo como hash em refresh_tokens.
  4. Define cookies rt (httpOnly) e csrf_token (não httpOnly).

6.2 Refresh

  • Frontend tenta automaticamente ao receber 401 (exceto bypass de auth).
  • Backend valida origem + CSRF em produção, valida refresh JWT, confere hash no banco, deleta token antigo e emite novo par.

6.3 Logout

  • Exige access token + origem confiável + CSRF em produção.
  • Revoga refresh token no banco e limpa cookies.

7. RBAC (controle por perfil)

  • CUSTOMER: reserva itens/slots, finaliza checkout, lista/vê próprios pedidos, cancela apenas PENDING.
  • VENDOR: CRUD itens do próprio tenant, gestão de inventário/slots, vê/atualiza pedidos do próprio tenant, não compra como cliente.
  • ADMIN: controle global de tenants/categorias/curadoria home, pode atuar em qualquer tenant.

8. Catálogo, Carrinho e Checkout

8.1 Catálogo (/api/items)

  • Filtros: type, categoryId, tenantId, search, locationId, geolocalização (lat/lng/radius), featuredHome, paginação.
  • Público/customer recebe dados públicos.
  • Vendor/admin recebem também dados operacionais.

8.2 Carrinho (soft reservation)

  • Reserva: POST /api/cart/reserve.
  • Sessão por cookie cart_session (uuid).
  • Produto físico: decrementa availableQty, incrementa reservedQty.
  • Serviço: slot AVAILABLE para RESERVED.
  • TTL da reserva: 15 minutos.
  • Limites anti-abuso por request/sessão e totais ativos.

8.3 Limpeza automática

Job por minuto remove reservas expiradas e devolve estoque/slot.

8.4 Checkout

  • Requer CUSTOMER autenticado.
  • Faz claim otimista das reservas no início da transação.
  • Separa pedidos por tenant (um checkout pode gerar múltiplos orders).
  • Consolida: físico baixa reservedQty, serviço RESERVED para BOOKED.
  • Persiste fulfillmentData criptografado.
  • Gera maintenanceIntake quando aplicável.

9. Pedidos e Status

Status possíveis:

PENDING -> APPROVED -> IN_PROGRESS -> COMPLETED

Regras:

  • Cancelamento permitido conforme matriz de transição.
  • Ao cancelar, devolve estoque físico e libera slots (BOOKED/RESERVED para AVAILABLE).
  • Fulfillment descriptografado e reduzido por role.
  • Email do usuário exposto apenas para admin.

10. Dashboard Operacional

Fonte: /api/vendor/kpis

  • KPIs: GMV, pedidos concluídos, total de itens, total de tenants (admin global).
  • Pedidos recentes para kanban e tabela.
  • Série temporal de GMV mensal (últimos 6 meses).
  • Curadoria da Home (admin): categorias/serviços destacados + ordem.

19. Troubleshooting

19.1 Erro de chave de criptografia

  • Mensagem: ORDER_DATA_ENC_KEY inválida.
  • Causa: valor não-base64 de 32 bytes.
  • Correção: gerar nova chave e atualizar env.

19.2 401 contínuo no frontend

  • Verificar JWT_SECRET, JWT_REFRESH_SECRET, cookies e origem.
  • Conferir se domínio/porta do frontend está em CORS_ORIGINS.

19.3 403 em refresh/logout/change-password (produção)

  • Causa comum: origem não permitida ou CSRF header ausente/incorreto.
  • Revisar Origin/Referer e X-CSRF-Token.

19.4 Itens sem aparecer no catálogo

  • Verificar filtros (type/category/tenant/location).
  • Conferir estoque (availableQty > 0) e slots AVAILABLE futuros.

19.5 Falha ao finalizar checkout

  • Pode ser disputa de concorrência (carrinho alterado durante claim).
  • Atualize carrinho e tente novamente.

19.6 Vendor sem acesso ao item/pedido

  • Escopo multi-tenant: vendor só acessa recursos do próprio tenantId.

20. Observações importantes de manutenção

  • frontend/README.md atual é template padrão Vite e não descreve o produto.
  • A política de senha no frontend (regex local) está mais permissiva que a política forte do backend; o backend prevalece.
  • Em produção, preferir prisma migrate deploy (já aplicado no compose de produção).
